<p>A checkpoint is the primary means for validating data in a production deployment of Great Expectations.</p>
<p>Checkpoints provide a convenient abstraction for bundling the Validation of a Batch (or Batches) of data against
an Expectation Suite (or several), as well as the Actions that should be taken after the validation.</p>
<p>A Checkpoint uses a Validator to run one or more Expectation Suites against one or more Batches provided by one
or more Batch Requests. Running a Checkpoint produces Validation Results and will result in optional Actions
being performed if they are configured to do so.</p>

<li><p><strong>name</strong> – User-selected cCheckpoint name (e.g. “staging_tables”).</p></li>
<li><p><strong>data_context</strong> – Data context that is associated with the current checkpoint.</p></li>
<li><p><strong>config_version</strong> – Version number of the checkpoint configuration.</p></li>
<li><p><strong>template_name</strong> – The name of another checkpoint to use as a base template.</p></li>
<li><p><strong>run_name_template</strong> – A template to create run names, using environment
variables and datetime-template syntax (e.g. “%Y-%M-staging-$MY_ENV_VAR”).</p></li>
<li><p><strong>expectation_suite_name</strong> – Expectation suite associated with checkpoint.</p></li>
<li><p><strong>batch_request</strong> – Batch request describing the batch of data to validate.</p></li>
<li><p><strong>action_list</strong> – A list of actions to perform after each batch is validated.</p></li>
<li><p><strong>evaluation_parameters</strong> – Evaluation parameters to use in generating this checkpoint.</p></li>
<li><p><strong>runtime_configuration</strong> – Runtime configuration to pass into the validator’s runtime configuration
